## 공부를 시작한 배경?

내가 취업을 시작할 즈음엔, `php`로 제작한 웹페이지들이 주를 이루었다.
`Javascript`의 라이브러리인 `JQuery`가 웹 개발의 주요 도구였고, 어느 순간에서부터 `React`, `Vue.js`, `Angular` 등의
모던 프레임워크와 라이브러리가 등장하면서 급변하던 시기였다.

그 중에서도`Facebook`에 의해 출시된 `React`는 선언적이고 컴포넌트 기반의
UI 라이브러리로, 가상 DOM을 도입하여 효율적인 렌더링 성능을 보여주며, 큰 인기를
끌고 주요 기술 스택으로 자리를 잡아 나 또한 흐름에 따라 `React` 사용을
선호하게되었다.

> Client-Side Rendering (CSR)의 유행

`React`, `Vue.js`, `Angular`와 같은 싱글 페이지 어플리케이션(SPA) 도구들은 클라이언트 사이드에서 주로 렌더링되며, 이 방식은 사용자 인터렉션에 대한 빠른 응답성을 보여줬고
현업에서 이러한 프레임워크 혹은 라이브러리에 대한 사용을 지향하였으며, 당시 채용 공고에도 이러한 기술 스택을 보유한 Front-end Engineer 들을 찾았다.

> `TypeScript`의 도입

자바스크립트의 슈퍼셋인 `Typescript`의 도입으로, 정적 타입 체크를 제공하여 대규모 뿐만 아니라 다양한 엔지니어들과 협업을 해야하는 프로젝트에서
관리와 버그 발견에 도움을 주었고, `Typescript`의 중요성은 날로 더해졌다.

1. **초기 로딩시간이 길다.**

- CSR 방식은 모든 자바스크립트를 로드하고 실행한 후에, 컨텐츠를 렌더링하기 때문에 초기 페이지 로딩 시간이 길어짐

2. **SEO 문제**

- 웹 크롤러들은 자바스크립트 실행을 제대로 처리하지 못해서, CSR 구현 방식 페이지는 검색엔진 최적화(SEO)에 문제가 생길 수 있음.

3. **서버 부담은 감소하나, 클라이언트 부담의 증가**

- 모든 렌더링 작업이 클라이언트 측에서 이루어지므로, 낮은 사양의 장치나 느린 네트워크 환경에서는 성능 이슈의 발생 가능성이 있음.

4. **보안 문제**

- 클라이언트 측에서 데이터를 처리할 떄, 민감한 데이터나 알고리즘을 클라이언트 코드에 포함하면 보안 문제가 생길 수 있음.

이 외에도 다양한 단점들이 있지만, 내가 생각하는 주요한 문제점들만 우선 나열해보았다.
나는 이러한 단점과 한계에 봉착하였고, SSR 방식을 사용하여 이를 극복하고자 하여 공부를 시작하게 되었다.

### Next.js의 등장 배경

- `React`는 CSR을 주로 사용하는 라이브러리였기 때문에, React 기반의 웹/앱 어플리케이션에서 SSR을 구현하기 위해서는 추가적인 설정과 코드가 필요하였고,
- 이러한 복잡성을 줄이기 위해, `vercel`이 `Next.js`를 개발하였다. `Next.js`는 `React` 기반의 프레임워크로, SSR을 간편하게 구현할 수 있게 도와준다.
- 또한, `Next.js`는 라우팅, 코드 분할, 정적 사이트 생성(SSG), API 라우트 등과 같은 다양한 기능을 제공하여, 웹 어플리케이션 개발의 생산성을 향상해준다.

결과적으로, 사용자 경험과 SEO의 중요성이 부각되면서 SSR의 필요성이 다시금 인식되었고, 이러한 배경 속에서 `Next.js`는 위의 단점들을 개선해주게 되며 큰 인기를 얻게 되었다.

### 그럼 이제 뭘 할까?

이 `Next.js`를 도대체 어떻게 활용해서, 어떤 웹 어플리케이션을 구축할까 고민하던 중, 아주 적합한 프로젝트를 찾았다.

**정적 페이지인 블로그를 만들어보면 어떨까?** 라는 생각에서 지금 보이는 이 블로그를 개발하게 되었고,
평소 벨로그나 티스토리 등을 통해 다양한 기술 블로그를 보며 언젠간 나도 기술 블로그를 써야지하며 차일피일 미뤄오던 것을 드디어 실천에 옮기게 되었다.
내가 손수 개발한 블로그에서 나의 기술 블로그를 작성하면, 아마 애착도 더 가지 않을까 싶어 이 블로그를 개발했고 탄생했다.

이 블로그를 어떤 방식으로 개발하였고, 구현하였는지에 대해 앞으로 차근차근 정리하며 포스팅을 시작할 예정이다.
